\chapter{相关技术综述}

\section{Spring Boot 2.6.13 后端开发框架}
在后端架构的选择上，本系统决定采用 Spring Boot 2.6.13 版本。这一选择的主要考量是其在当前开发生态中的稳定表现，以及其特有的“约定优于配置”理念对开发效率的显著提升。通过引入起步依赖（Starters）和自动配置机制，笔者在开发过程中能够绕过繁琐的环境部署与依赖配置阶段，从而更直接地投入到业务逻辑的实现中\cite{Yang2021, Zhao2020}。

针对科研经费多维分析的复杂需求，Spring Boot 的服务编排能力表现得非常关键。在本系统的 \texttt{FundStatisticsServiceImpl} 模块里，系统利用其微内核架构，对项目、预算、支出等核心数据实体进行了深度的动态聚合。在处理海量经费记录时，后端能够高效地将底层数据转换为结构化的多维分析对象（如 \texttt{ExecutionRateVO}）。此外，内嵌的 Tomcat 容器配合异常处理机制，也能够为高并发下的数据调取提供稳健的后台保障，这在很大程度上确保了决策支持功能在统计查询场景下的响应速度。

\section{MyBatis-Plus 3.5.3.1 持久层增强工具}
就数据持久化的处理而言，本系统引入了 MyBatis-Plus 作为增强工具。它的设计宗旨是在保留 MyBatis 原生灵活性的基础上，通过内置的 BaseMapper 自动实现大量的 CRUD 操作。考虑到本系统选用的 3.5.3.1 版本提供了成熟的 Lambda 条件构造器，开发中可以利用更直观的面向对象方式来构建 SQL，这不仅降低了拼写错误的风险，也让持久层的代码逻辑变得更加清晰。

MyBatis-Plus 的引入对于处理多维分析所需的数据聚合场景尤其管用。传统的做法往往需要编写大量的 XML 映射文件，而 MP 提供的 LambdaQueryWrapper 使得诸如“按类别统计”或“计算总体执行率”等操作变得非常简洁。不仅如此，系统还利用其自动填充特性来管理经费流向的审计记录，确保每一笔支出的生命周期都能被准确追踪。配合其分布式 ID 生成策略，系统为 \texttt{FundStatisticsSnapshot} 统计快照提供了可靠的存储支持，从而为历史趋势分析打下了坚实的数据基础。

\section{Vue 3.5.12 与 ECharts 5.5.1 可视化技术}
在前端展现层面，Vue 3 是目前构建响应式界面的主流工具。本系统所采用的 3.5.12 版本通过组合式 API（Composition API），在逻辑复用和运行时性能上比前代有了明显改进。而为了让决策支持看板更加直观，本研究选取了 ECharts 5.5.1 作为可视化引擎。它具备 Canvas 和 SVG 的双引擎渲染能力，即使面对海量的财务数据点，也能在前端实现流畅的动态呈现\cite{Xie2019}。

在构建实际的监控仪表盘时，Vue 3 与 ECharts 的结合实现了一种从静态图表到动态交互的跨越。通过 Vue 3 的响应式状态管理，系统能够实时捕捉用户在多维看板上的各种筛选动作，比如按二级学科或时间窗口进行的视角切换。在这种模式下，ECharts 可以在毫秒级内完成重绘，将复杂的统计数据转化为易于观察的执行趋势线或科目占比图。这种深度交互的视觉体验，将原本枯燥、碎片化的数字转化成了具备直觉指引性的决策依据。

\section{本章小结}
通过对系统整体技术架构的梳理，本章详细探讨了由 Spring Boot、MyBatis-Plus 以及 Vue 3/ECharts 构成的全栈技术链条。其中，Spring Boot 与 MyBatis-Plus 共同支撑起了从底层数据持久化到业务聚合的高速处理逻辑；而 Vue 3 与 ECharts 则在表层实现了多维模型到可视化看板的动态映射。这些技术的融合，不仅实现了数据“采集-存储-分析-可视化”的逻辑闭环，也为后续章节关于系统设计与实现的讨论提供了稳固的技术基础。
